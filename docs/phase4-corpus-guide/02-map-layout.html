Step 2 - Feature Map, Scoring, And Layout

Prepare the private feature contract used by every capture row.

Previous: Private Intake Overview Next: Capture Export

Goal

This step creates the private files that define what features to extract and how to score them. The checked-in demo map and scoring program are placeholders and must not be treated as golden evidence.

Files To Place In The Bundle

File Purpose
feature-map.yaml Real regions, offsets, feature semantics, stability, and discretization hints.
scoring-program.yaml Staged first-boss milestones, goal predicate, and prune rules.
workload-image.yaml or workload-image-ref.txt WorkloadImage provenance, validation stamp, framebuffer format, and pad layout.
layout.json Compiled byte layout used to pack every feature_bytes record.

Keep real offsets and scoring labels private unless the operator has explicitly approved publication. Put them in the private bundle, not in feature-maps/demo-game.yaml or scoring/demo-game.yaml.

Validate Feature Map And Scoring Program

cargo run --locked -p refwork-featuremap -- validate \
  <private-bundle-dir>/feature-map.yaml \
  --scoring <private-bundle-dir>/scoring-program.yaml

Generate Layout Evidence

The capture-spec hash should identify the exact capture/export contract used for this corpus. It can be a BLAKE3 hash or an opaque private ref. 

cargo run --locked -p refwork-verify -- phase4-layout \
  --map <private-bundle-dir>/feature-map.yaml \
  --out <private-bundle-dir>/layout.json \
  --capture-spec-hash <hash-or-ref>

Acceptance Checklist

  • The map/scoring validation command exits successfully.
  • layout.json exists in the private bundle.
  • layout.json.total_len is greater than zero.
  • The layout hash will be copied into capture rows and the manifest.
  • The map has no placeholder offsets.
