# Review: ralph/iteration-9-implement-image-double-build-and-register-guard

## Findings

1. **High: `image double-build` can certify dirty source while stamping clean `HEAD`.**  
   `double_build` records `source_rev = git_rev(workspace_root)` and only enforces a clean `control-plane` checkout; it never checks the reference-workload checkout for modifications before materializing the clean roots ([xtask/src/image.rs:288](../../../../../xtask/src/image.rs#L288), [xtask/src/image.rs:291](../../../../../xtask/src/image.rs#L291)). The materializer then runs `git ls-files -z` but copies bytes from the live worktree (`source.join(rel)`) rather than from `HEAD` ([xtask/src/image.rs:397](../../../../../xtask/src/image.rs#L397), [xtask/src/image.rs:420](../../../../../xtask/src/image.rs#L420), [xtask/src/image.rs:422](../../../../../xtask/src/image.rs#L422)). That means any unstaged or staged modification to a tracked source file is included in both clean-root builds, the byte comparison passes, and the manifest still says it was built from the committed `git_rev` passed into `build_image_with_git_rev` ([xtask/src/image.rs:233](../../../../../xtask/src/image.rs#L233), [xtask/src/image.rs:239](../../../../../xtask/src/image.rs#L239)). This defeats the clean-root provenance contract: the reported commit no longer reproduces the artifacts. Either refuse a dirty source checkout before materialization or materialize from the recorded tree object, for example `git archive HEAD`, so the copied bytes and stamped revision are the same source.

2. **High: green-stamp acceptance is presence-only and can be bypassed with an empty file.**  
   `register_image` treats any existing `determinism.last_green` as a valid stamp and immediately returns `DirectDistStamped` ([xtask/src/image.rs:347](../../../../../xtask/src/image.rs#L347), [xtask/src/image.rs:353](../../../../../xtask/src/image.rs#L353)). `validate_handoff_files` likewise accepts the bundle if either `determinism.unstamped.yaml` or `determinism.last_green` exists, without parsing or binding the green stamp to the manifest hash, reference-workload rev, guest-sdk rev, suite version, timestamp, or report hash ([xtask/src/image.rs:1274](../../../../../xtask/src/image.rs#L1274), [xtask/src/image.rs:1276](../../../../../xtask/src/image.rs#L1276)). Package 06 explicitly says the stamp/report must carry suite version, git rev, timestamp metadata, and report hash, and register must refuse without a fresh green suite report ([.agents/plans/guest-sdk-unblock-reference-workload/06-full-determinism-suite.md:73](../../../../../.agents/plans/guest-sdk-unblock-reference-workload/06-full-determinism-suite.md#L73), [.agents/plans/guest-sdk-unblock-reference-workload/06-full-determinism-suite.md:124](../../../../../.agents/plans/guest-sdk-unblock-reference-workload/06-full-determinism-suite.md#L124)). With the current guard, `touch dist/.../determinism.last_green` is enough to pass `image register --require-green-stamp`, so the command does not enforce the contract it advertises.

3. **Medium: the automatic package-06 cutover is tied to the M5 evidence note rather than package-06 stamp support.**  
   `green_stamp_support_landed` flips unstamped registration off when `.agents/plans/guest-sdk-unblock-reference-workload/m5-suite-evidence.md` exists ([xtask/src/image.rs:544](../../../../../xtask/src/image.rs#L544), [xtask/src/image.rs:546](../../../../../xtask/src/image.rs#L546)). The package-04 contract says `image register` may be a direct handoff until control-plane registration exists, but must refuse missing stamps once package 06 lands ([.agents/plans/guest-sdk-unblock-reference-workload/04-image-handoff-assets.md:47](../../../../../.agents/plans/guest-sdk-unblock-reference-workload/04-image-handoff-assets.md#L47)). The closeout plan names `m5-suite-evidence.md` as a report-hash/green-stamp evidence note, not as the implementation signal that package-06 registration support is present ([.agents/plans/guest-sdk-unblock-reference-workload/07-ci-evidence-closeout.md:37](../../../../../.agents/plans/guest-sdk-unblock-reference-workload/07-ci-evidence-closeout.md#L37)). This creates a brittle milestone coupling: adding the evidence note can make package-04 direct handoff refuse unstamped bundles before a real stamp parser/register path exists, while a garbage `determinism.last_green` still passes because of finding 2.

4. **Medium: the M4 evidence note is not reproducible from the branch head it is committed on.**  
   The note says the branch is `ralph/iteration-9-implement-image-double-build-and-register-guard` but records the implementation rev as `3068fc815a3b9161acf308874bd55054e71f5a37` ([.agents/plans/guest-sdk-unblock-reference-workload/m4-image-handoff-evidence.md:18](../../../../../.agents/plans/guest-sdk-unblock-reference-workload/m4-image-handoff-evidence.md#L18), [.agents/plans/guest-sdk-unblock-reference-workload/m4-image-handoff-evidence.md:19](../../../../../.agents/plans/guest-sdk-unblock-reference-workload/m4-image-handoff-evidence.md#L19)). The actual branch head is the later evidence commit `371a50fba3ead1601f0d2326184a58daf5a72d5a`; rerunning `image build` or `image double-build` at that head changes `meta.built_from.git_rev` and therefore changes the recorded manifest and sidecar hashes ([xtask/src/image.rs:233](../../../../../xtask/src/image.rs#L233), [xtask/src/image.rs:846](../../../../../xtask/src/image.rs#L846), [.agents/plans/guest-sdk-unblock-reference-workload/m4-image-handoff-evidence.md:41](../../../../../.agents/plans/guest-sdk-unblock-reference-workload/m4-image-handoff-evidence.md#L41), [.agents/plans/guest-sdk-unblock-reference-workload/m4-image-handoff-evidence.md:61](../../../../../.agents/plans/guest-sdk-unblock-reference-workload/m4-image-handoff-evidence.md#L61)). Earlier evidence notes handle this by separating the checked source rev from the evidence-note commit; this one does not, so the durable M4 evidence is ambiguous for anyone trying to reproduce it from the reviewed branch.
